Built around the shot, not the software

Everything a stop-motion set already needs

FrameLabs was designed against a full feature specification, animator-first. A few of the fourteen features that make it up:

Live Camera View

Camera feed, onion skin, histogram, and safe areas together, holding 30+ FPS while you frame the shot.

One-Press Capture

Space bar to shoot. Frame appears in the timeline within a second, metadata and thumbnail already generated.

Onion Skin

Adjustable previous/next frame counts, opacity falloff, and tint color — matched frame to frame, not just adjacent.

Autosave & Crash Recovery

Silently autosaves after every capture, never interrupting a shot. Crash recovery on restart is next up.

Timeline

Select, replace, duplicate, and delete frames with full undo — 100 actions minimum, always reversible.

Open in Blender

One button. Camera, timeline, image sequence, and compositor are generated for you — no manual setup, ever.

01

Project Management

PARTIAL▾

New Project walks through name, folder, FPS, and resolution, then creates the project folder structure. Open Project restores the timeline, settings, metadata, overlays, and export history — and flags any missing frames instead of failing silently.

Create project workflow
File → New Project↓
Choose Project Name↓
Choose Folder↓
Choose FPS↓
Choose Resolution↓
Create → project.ffproj + images/, thumbnails/, cache/, exports/, metadata/
Acceptance criteria
  • Project folder and initial project file are created
  • Timeline starts empty; autosave starts immediately
  • Window title displays the project name
Edge cases
  • Folder already exists · invalid filename · no write permission · disk full
  • Open Project with missing images warns "14 frames are missing" and offers Continue, Locate Missing Files, or Cancel
02

Camera Manager

PARTIAL▾

Detects cameras on startup and handles hot-plugging and disconnects gracefully — but right now that's webcams only. DSLR (gphoto2) and libcamera backends aren't built yet.

Detect cameras
No Camera→Scanning...→"Canon EOS R50 Connected"
Acceptance criteria
  • Hot-plugging supported; disconnects handled gracefully
  • Current frame finishes if possible when a camera drops mid-capture
Currently
  • Webcam backend is real and complete — connect/disconnect, capture, live view, hot-plug detection, "Camera Lost" reconnect/retry, all verified
  • DSLR (gphoto2) and libcamera backends don't exist yet — every backend implements the same Camera interface, so adding them won't touch the rest of the app

Live View

Camera feed, overlays, onion skin, histogram, and safe areas in one view, targeting 30+ FPS. Zoom on the wheel, pan on middle-mouse, fit with F.

Controls
Scroll Zoom
Middle-click Pan

Acceptance criteria
  • Feed, overlays, onion skin, histogram, and safe areas render together

Currently
  • Camera feed, zoom/pan/fit, onion skin, playback, safe area guides, and the histogram overlay are all closed and verified — Feature 3 is fully complete
04

Capture

DONE▾

Space or the capture button triggers the camera, verifies and writes the image, extracts metadata, generates a thumbnail, and updates the timeline — with no duplicate frame numbers and no corrupt frames on a timeout.

Capture workflow
Trigger Camera→Receive Image→Verify Image→Write PNG→Extract Metadata→Generate Thumbnail→Update Timeline→Autosave→Notify Blender→Ready
Acceptance criteria
  • Frame appears within one second, hardware permitting
  • Timeline updates immediately; no duplicate frame numbers
Failure cases
  • Camera timeout → "Capture Failed / Retry" — no corrupt frame is ever created
  • Disk full → "Disk Full / Capture Aborted" — the project stays usable
  • Write failure retries once, then shows an error
05

Timeline

DONE▾

Real thumbnails, click-to-select, click-and-drag scrolling, and marker/selection borders, plus Delete, Replace, Duplicate, Marker, and Notes — all wired into a right-click menu, a selection action bar, and keyboard shortcuts, all undoable. Reordering is planned for a future release.

Replace frame workflow
Select Frame→Replace→Capture New Image→Overwrite Existing→Refresh Thumbnail
Acceptance criteria
  • Frame number stays the same after a replace
  • Delete asks "Delete Frame 152?" and is always undoable
Currently
  • The real TimelineWidget is built: thumbnails, click-to-select driving the playhead, and independent marker/selection borders that can stack on one frame
  • Click-and-drag scrolling on the timeline strip, distinguishing a real drag from a small-movement select-click so selecting a frame still works
  • Delete, Replace, Duplicate, Marker toggle, and Notes are all wired in — through a right-click context menu, a selection-triggered action bar docked below the Timeline strip, and keyboard shortcuts, sharing one set of handler methods behind every surface, each action backed by a real, undoable Command
  • Delete, Replace, and Duplicate back up the real image/thumbnail/metadata files they touch before acting, so undo restores the exact original files, not just an in-memory record
  • Drag-to-reorder frames is spec'd but intentionally deferred to a future release
06

Onion Skin

DONE▾

Adjustable previous/next frame counts, opacity, and tint colors, tuned to stay out of the way of live preview performance on modern hardware.

Adjustable controls
Opacity·Previous Frame Count·Next Frame Count·Tint Color
Acceptance criteria
  • Supports both previous and next frames, not just adjacent ones
  • No noticeable impact on live preview performance on modern hardware
07

Playback

DONE▾

Play, pause, and loop at 25/50/100/200% speed, with live speed changes taking effect on the next frame. Playback never modifies the project — reverse playback is planned for later.

Controls
Play·Pause·Loop·Reverse (future)
Speed: 25% · 50% · 100% · 200%
Acceptance criteria
  • Playback should never modify the project
Currently
  • Real worker-thread playback, manually verified alongside Onion Skin — including a shutdown-timing crash that surfaced once both ran together, now fixed and verified
  • Reaching the end of a non-looping sequence now resets the playhead to frame 1 automatically, so the timeline is immediately ready to play again with no manual reselection
08

Autosave

PARTIAL▾

Autosaves silently after every capture. The independent 30-second timer and crash recovery on restart are spec'd but not built yet.

Trigger
Every 30 secondsANDafter every capture
Acceptance criteria
  • Silent — never interrupts a capture in progress
  • Keeps the last 20 autosaves
Currently
  • After-every-capture autosave is real and wired into the capture pipeline
  • The independent 30-second timer and the "Recovered Project Found — Restore? / Discard?" crash recovery flow don't exist yet
09

Undo / Redo

PARTIAL▾

Real Command/UndoManager infrastructure with a 100-action history and full redo. Delete, Replace, Duplicate, Marker, and Notes are all undoable now — the original Capture action and project settings changes aren't wrapped in Commands yet.

Covers
Capture·Delete·Replace·Notes·Timeline edits·Project settings
Acceptance criteria
  • Minimum 100-action history
  • Redo supported for everything undo covers
Currently
  • Delete, Replace, Duplicate, Marker toggle, and Notes edits are each a real Command, wired to Ctrl+Z / Ctrl+Shift+Z and the Edit menu, with a 100-action history
  • Replace's Command runs on the capture worker thread since it triggers a real camera capture, and is only recorded on the undo stack after it succeeds — redo reapplies the captured result rather than firing the camera again
  • The original Capture action and Project Settings changes aren't wrapped in Commands yet, so neither is undoable yet
10

Blender Export ★

NOT STARTED▾

Saves the project, generates export data, launches Blender, and runs a script that imports the sequence, sets up the camera and timeline, and saves the scene — ready to open, no manual setup.

Open in Blender workflow
Save Project→Generate Export Data→Launch Blender→Run Blender Script→Import Sequence→Configure Camera→Set Timeline→Save Blend→Open Scene
Acceptance criteria
  • Blender scene auto-creates Camera, Timeline, Image Sequence, Compositor, Output Settings, and Project Metadata
  • No manual setup required in Blender
Failure cases
  • Blender not installed → prompts to locate the executable, then remembers it
  • Blender already running → offers Reuse Existing Blender or Open New Instance

Live Blender Sync (v2)

NOT STARTED▾

With sync enabled, every capture sends the new frame, timeline update, and metadata straight into an already-running Blender scene — no export, no reload button.

When enabled
Capture→New Frame + Timeline Update + Metadata→Blender updates automatically
Acceptance criteria
  • No export step and no reload button required
12

Keyboard Shortcuts

DONE▾

Every shortcut is reassignable via config.json. Defaults include Space to capture, Ctrl+S to save, B to open Blender, O to toggle onion skin, and Delete to delete a frame.

Defaults
Space Capture
Ctrl+S Save
Ctrl+Z Undo
Ctrl+Shift+Z Redo
Ctrl+D Duplicate frame
Delete Delete frame
B Open Blender
O Toggle onion skin
← / → Previous / next frame
Enter Play / stop
Acceptance criteria
  • Every shortcut above is reassignable
Currently
  • Every default above reads from Config's keyboard_shortcuts setting through a Qt-free parse_shortcut() helper — genuinely reassignable by editing ~/.framelabs/config.json, not just hardcoded QKeySequence defaults
  • No in-app settings UI to change a shortcut yet — deliberately deferred, since the Feature Spec only requires configurability, not a specific UI for it
13

Plugin Manager

PARTIAL▾

Scans a plugins/ folder on startup and loads only valid plugins. A crashing plugin is disabled automatically — the rest of the app keeps running.

Startup workflow
Startup→Search plugins/→Load valid plugins only
Acceptance criteria
  • A plugin exception disables that plugin, not the app
  • The application continues running normally
Currently
  • Discovery, isolated loading, and startup wiring are closed and verified — a bad plugin, a plugin missing its interface, or a plugin whose activation fails is caught and logged without taking down the app
  • Still ahead: no plugin management UI yet, and no deactivate/event-bus hooks for a plugin to actually act on capture, timeline, or camera events

Logging

DONE▾

Every meaningful operation — captures, autosaves, Blender launches, plugin failures — is logged, in past tense, to make bug reports reproducible.

Examples logged
Project Opened·Capture Started·Capture Complete·Autosave·Blender Launch·Plugin Loaded·Plugin Failed
Acceptance criteria
  • Every important action is logged, in past tense
  • Logs are detailed enough to reproduce a bug report

Developer handbook

Built to be understood, not just used

FrameLabs' Developer Handbook is the source of truth for implementation decisions — if code conflicts with it, the handbook wins. Here's the shape of it.

Guiding principles

Every design decision reinforces these.

01

Never Lose User Data

A captured frame is the most valuable thing in the app. Nothing — a crash, a disconnect, a failed export — should ever lose one. If there's uncertainty, preserve the data.

02

UI Never Blocks

Saving, importing, exporting, thumbnailing, and launching Blender all run on background workers. The interface stays responsive, always.

03

Small Modules

Each module does one thing. CaptureService, not CaptureManagerThatAlsoHandlesUISavingExportsAndPlayback.

04

Explicit Over Clever

Readable beats clever. No unnecessary abstraction, no hidden behavior — clarity first.

05

Composition Over Inheritance

Objects are assembled from smaller services rather than built through deep inheritance hierarchies.

System architecture

UI never touches hardware or the filesystem directly — everything routes through services and the event bus.

Camera interface

Every backend — webcam, gphoto, libcamera — implements the same contract. The rest of the app never knows which is connected.

  • connect() ()
  • disconnect()
  • capture()
  • start_live_view()
  • stop_live_view()
  • set_iso()
  • set_shutter()
  • set_aperture()
  • get_metadata()

Capture pipeline

A fixed sequence — never reordered without justification.

  • Trigger camera
  • Receive image
  • Verify image
  • Write image + metadata
  • Generate thumbnail
  • Update timeline
  • Autosave
  • Emit FRAME_CAPTURED

Event bus

Modules talk through events, named in past tense — they describe something that already happened.

FRAME_CAPTURED PROJECT_OPENED PROJECT_SAVED TIMELINE_UPDATED CAMERA_CONNECTED CAMERA_DISCONNECTED PLAYBACK_STARTED BLENDER_SYNC

Project file format

Projects are plain folders. No proprietary binary formats — project.ffproj is readable, versioned JSON.

{
  "version": 1,
  "name": "Robot Walk Cycle",
  "fps": 12,
  "resolution": [6000, 4000],
  "camera": { "model": "Canon EOS R50", "lens": "50mm" },
  "frames": [ { "number": 1, "file": "images/000001.png" } ]
}
